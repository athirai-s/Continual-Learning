# Training Flow

This document describes the end-to-end training pipeline as configured in
`train_colab.ipynb`, including how the augmented dataset was created and how it
flows through training and evaluation.

---

## Overview

Training is a **continual learning** experiment: a language model is trained
sequentially across four temporal snapshots of Wikipedia, one period at a time.
After each period the model is evaluated, and a checkpoint is saved to Google
Drive so runs can be resumed across Colab sessions.

```
Periods (in order):  aug_sep → sep_oct → oct_nov → nov_dec
```

The full pipeline is:

```
Augmented CSVs (data/augmented/TWiki_Diffsets/)
        ↓
build_augmented_dataset()     ← patches TemporalWikiDataset in-place
        ↓
CASFTrainer.train_period()    ← one period at a time
        ↓
trainer.checkpoint()          ← saved to Google Drive after each period
        ↓
run_period_evaluation()       ← plasticity / stability / token_f1
```

---

## The Augmented Dataset

### Why it exists

The original dataset (`TWiki_Diffsets.zip`) contains raw Wikipedia diff passages.
These are long, noisy, and contain far more content than is needed to teach any
individual fact — training on them produced very slow learning signal.

The augmented dataset replaces each passage with a **short, focused passage
generated by Gemini** (specifically `gemini-2.5-flash-lite`).  Each generated
passage is 1–3 sentences, contains exactly the fact stated in the corresponding
eval probe, and includes brief topical context.  This makes the learning signal
much tighter and better matched to the evaluation task.

### How it was created

The source of truth for facts is the eval probe set in `TWiki_Probes.zip`.  The
generation pipeline was:

1. **`dataset_utils/export_eval_prompts.py`** — reads all `changed` and
   `unchanged` probes from `TWiki_Probes.zip` and writes them as
   `<cloze_prompt>\t<ground_truth>` lines to `dataset_utils/prompts/`.  Probes
   are written in order: all `changed` probes first, then all `unchanged`
   probes.

2. **`dataset_utils/generate_dataset.py`** — reads those prompt files, calls
   Gemini once per `(prompt, ground_truth)` pair, and writes the resulting
   passages to `data/augmented/TWiki_Diffsets/<period>.csv` (one CSV per
   period).  The row order in each CSV matches the prompt order exactly —
   changed-probe passages first, then unchanged-probe passages.

The four CSVs are committed to the repo under `data/augmented/TWiki_Diffsets/`:

```
aug_sep.csv   sep_oct.csv   oct_nov.csv   nov_dec.csv
```

Each file has a single `text` column.  Rows that failed generation are written
as `ERROR` and are skipped at load time.

### Correspondence between CSV rows and probes

Because the generation order mirrors the probe export order, the row index in a
CSV maps directly to a probe index:

| CSV rows | Probe split |
|---|---|
| `0 .. N_changed - 1` | `changed` probes `0 .. N_changed - 1` |
| `N_changed .. N_changed + N_unchanged - 1` | `unchanged` probes `0 .. N_unchanged - 1` |

This correspondence is what makes `dataset_fraction` (see below) safe to use —
selecting a prefix of each block keeps training and evaluation on exactly the
same facts.

## Notebook Walkthrough

The notebook (`train_colab.ipynb`) is the single entry point for running
experiments.  All configuration lives in the first code cell; the remaining
cells are run top-to-bottom without editing.

### 1. Configuration cell

Key parameters:

| Variable | Purpose |
|---|---|
| `METHOD` | Training method: `full_ft`, `lora`, `smf`, or `casm` |
| `MODEL_NAME` | HuggingFace model ID (must have approved access) |
| `RUN_ID` | Unique name for this experiment; checkpoints are stored under it |
| `PERIODS` | Subset of `["aug_sep", "sep_oct", "oct_nov", "nov_dec"]` to train |
| `DRIVE_DIR` | Google Drive path for checkpoints and cache |
| `USE_AUGMENTED_DATA` | `True` → use committed augmented CSVs; `False` → use `TWiki_Diffsets.zip` |
| `DATASET_FRACTION` | Fraction of data to use (see below) |
| `BATCH_SIZE` / `GRAD_ACCUM_STEPS` | Effective batch size = product of both |
| `EPOCHS_PER_PERIOD` | How many full passes over each period's passages |
| `MAX_PASSAGES_PER_PERIOD` | Hard cap on passages (applied after fraction) |
| `CAPTURE_ACTIVATIONS` | Whether to save per-layer hidden states after each period |

Method-specific blocks (`LORA_*`, `SMF_*`, `CASM_*`) are only used when the
matching `METHOD` is selected.

### 2. Google Drive mount and directory setup

Mounts Drive and creates:
- `CHECKPOINT_DIR = DRIVE_DIR/checkpoints` — model + optimizer state
- `DATASET_CACHE_DIR = DRIVE_DIR/dataset_cache` — unused by the augmented path,
  kept for compatibility

### 3. Repo clone / pull

Clones `https://github.com/athirai-s/Continual-Learning` into `/content/Continual-Learning`
(or pulls if already present from a previous session).

### 4. Dependency install

Installs pinned versions of `transformers`, `peft`, `trl`, and `datasets` that
are not pre-installed on Colab, then adds the repo to `sys.path`.

### 5. Dataset verification

Asserts that the required files exist before attempting any network calls:

- `USE_AUGMENTED_DATA=True` → checks `TWiki_Probes.zip` (bundled in repo) and
  all `<period>.csv` files for the configured `PERIODS`
- `USE_AUGMENTED_DATA=False` → checks `TWiki_Diffsets.zip` and `TWiki_Probes.zip`
  in `data/`

### 6. HuggingFace authentication

Prompts for a HuggingFace token with read access to the model (required for
gated models like Llama).

### 7. Augmented CSV verification (training cell pre-check)

Re-verifies each period's CSV immediately before building the config, and prints
size in bytes.  Fails with a clear message if any CSV is missing.

### 8. TrainConfig construction

Builds a `TrainConfig` dataclass from all configuration variables and calls
`cfg.validate()`, which catches invalid settings before any model is loaded.
Prints a summary of the resolved config.

### 9. Checkpoint resume detection

Looks for `CHECKPOINT_DIR/<RUN_ID>/latest.json`.  If found, sets `RESUME_FROM`
so `run_training()` will skip already-completed periods and resume mid-period
if training was interrupted.

### 10. Training (`run_training`)

Calls `run_training()` from `training/train_runner.py` with:

- **`model_factory`** — loads the backbone and wraps it for the chosen method
  (LoRA adapter, SMF memory module, or CASM slot bank)
- **`dataset_factory`** — `build_augmented_dataset` if `USE_AUGMENTED_DATA`,
  otherwise `build_real_dataset`
- **`checkpoint_dir`** — the Drive path
- **`training_units`** — the configured `PERIODS` list

For each period the runner:

1. Calls `dataset_factory(period, cfg)` → returns a patched `TemporalWikiDataset`
2. Calls `trainer.train_period(dataset, period)` → trains for `EPOCHS_PER_PERIOD`
3. Saves a checkpoint to `CHECKPOINT_DIR/<RUN_ID>/`
4. If `eval_after_each_period=True`, calls `run_period_evaluation()` and records
   `plasticity`, `stability`, and `token_f1` for both the `changed` and
   `unchanged` probe splits
5. If `CAPTURE_ACTIVATIONS=True`, saves per-layer hidden states alongside the
   eval outputs

### 11. Results summary

Prints a table of final training loss, passage count, train time, and eval
metrics for each period.

## `DATASET_FRACTION`

`DATASET_FRACTION` controls what share of the dataset is used for **both
training and evaluation**, keeping them on exactly the same facts.

```python
DATASET_FRACTION = None    # use everything (default)
DATASET_FRACTION = 0.1     # use the first 10% of each split
DATASET_FRACTION = 0.25    # use the first 25% of each split
```

The fraction is applied **proportionally and independently** to the `changed`
and `unchanged` probe splits:

```
k_changed   = ceil(N_changed   * fraction)
k_unchanged = ceil(N_unchanged * fraction)
```

Training passages are then drawn from the CSV rows that correspond to those
probes (the first `k_changed` rows, plus `k_unchanged` rows starting at
`N_changed`).  This preserves the index alignment described in the
[correspondence table](#correspondence-between-csv-rows-and-probes) above.

The value is included in the resume compatibility check, so attempting to resume
a run with a different `DATASET_FRACTION` will fail fast with a clear error.

## How `build_augmented_dataset` Works

`build_augmented_dataset` (in `training/train_runner.py`) returns a standard
`TemporalWikiDataset` object with its data-loading methods monkey-patched so
that:

- **`get_train_passages()`** returns the selected passages from the augmented
  CSV instead of reading `TWiki_Diffsets.zip`
- **`_load_probes(split)`** is replaced with a no-op when `dataset_fraction` is
  active (the probe subsets are pre-loaded and stored directly in
  `dataset._probes`)
- **`TWiki_Probes.zip`** is pointed at the copy bundled in the repo
  (`casf_dataset_api/download_dataset_scripts/data/TWiki_Probes.zip`) so no
  separate download is required

The rest of the training and evaluation machinery interacts with the dataset
through its standard interface (`load()`, `get_probes()`,
`get_train_passages()`), with no awareness of the patching.
